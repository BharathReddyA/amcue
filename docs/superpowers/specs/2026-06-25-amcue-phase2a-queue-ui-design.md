# AMcue Phase 2a — Functional Queue/Feed UI (stubbed content)

Date: 2026-06-25
Status: Approved

## Purpose

Phase 2 of the AMcue design (AI core) is split into two parts so the team can see and
click through a real, functional UI before any AI integration work happens:

- **Phase 2a (this spec):** build the actual Generate → Queue → Approve/Reject → Feed
  loop end-to-end, using a stub content generator instead of a real AI call.
- **Phase 2b (separate, later spec):** swap the stub for real Gemini calls behind the
  same interface. No UI changes required — only the generator implementation changes.

This lets the team demo and react to the real interaction loop immediately, while AI
integration (API keys, prompt design, error handling for a third-party service) is
deferred to its own focused unit of work.

## Scope boundaries

- No real AI calls in this phase. Content is generated by a stub function.
- "Reject" simply removes an item from the queue — there is no rejected-items view (per
  the original Phase 3 design: rejected content just doesn't surface anywhere).
- No styling/polish pass — that's Phase 4 per the original design. Plain HTML elements
  are fine, consistent with Phase 1.

## Stub generator interface

`backend/src/services/ai/stubProvider.js`:

```js
function generateStubContent(project) {
  return {
    caption: `Check out ${project.name} — ${project.description}!`,
    imageUrl: `https://picsum.photos/seed/${project.id}/600/400`,
  };
}

module.exports = { generateStubContent };
```

Deterministic per project (seeded by `project.id`) so repeated generates for the same
project don't reroll the placeholder image, keeping demo behavior predictable. Phase 2b
will add `backend/src/services/ai/geminiProvider.js` implementing the same shape
(`{ caption, imageUrl }` returned from a function taking a project), and the route will
swap which provider it imports — no other code changes.

## API routes (backend)

Both new routes require `requireAuth` and must scope to the requesting user's own data
(same pattern as existing project routes).

- `POST /projects/:id/generate`
  - Verifies the project belongs to `req.userId` (404 if not, reusing the existing
    ownership-check pattern from `GET /projects/:id`).
  - Calls `generateStubContent(project)`.
  - Creates a `ContentItem` with `appProjectId`, `caption`, `imageUrl`, `status: 'pending'`.
  - Returns the created `ContentItem` (201).

- `GET /projects/:id/content?status=pending|approved`
  - Returns `ContentItem` rows for that project filtered by `status` query param.
  - 400 if `status` is missing or not one of `pending`/`approved`.
  - 404 if the project doesn't belong to `req.userId`.

- `PATCH /content/:id`
  - Body: `{ status: 'approved' | 'rejected' }`. 400 if status is anything else.
  - Looks up the `ContentItem`, joins to its `AppProject` to verify ownership via
    `req.userId` (404 if the item doesn't exist or isn't owned by the requester).
  - Updates `status`, returns the updated `ContentItem`.

## Frontend pages

- `frontend/app/projects/[id]/page.js` — project detail page. Shows name, description,
  screenshot thumbnails (if any), a "Generate content" button (calls
  `POST /projects/:id/generate`, then refetches), and links to `/queue` and `/feed`.
- `frontend/app/projects/[id]/queue/page.js` — fetches
  `GET /projects/:id/content?status=pending`. Each item renders the image, caption, and
  Approve/Reject buttons. Approve → `PATCH /content/:id` with `approved` → item removed
  from local list. Reject → same with `rejected` → item removed from local list.
- `frontend/app/projects/[id]/feed/page.js` — fetches
  `GET /projects/:id/content?status=approved`. Read-only list (image + caption, no
  actions).

All three pages follow the existing pattern from `frontend/app/projects/page.js`: client
component, `isLoggedIn()` redirect guard, `apiFetch` for data, inline error display.

## Data flow

1. User clicks "Generate content" on the project detail page.
2. Backend creates a `pending` `ContentItem` with stub caption/image.
3. User navigates to `/queue`, sees the pending item.
4. Approve → item's status becomes `approved` → vanishes from `/queue`, now appears on
   `/feed`. Reject → status becomes `rejected` → vanishes from `/queue`, never shown
   anywhere else.

## Error handling

Consistent with the existing codebase pattern (Task 5 of Phase 1): routes wrap their
core logic, return clear 4xx for bad input/ownership failures, and a 5xx with a generic
message for unexpected failures (DB errors). No special handling needed for the stub
generator since it has no I/O and cannot fail.

## Testing approach

Same minimal, practical approach as Phase 1: a handful of backend route tests
(supertest) covering generate → creates a pending item, list-by-status filtering, and
approve/reject transitions — enough to catch regressions as the team builds Phase 2b on
top of these routes. No frontend test suite, consistent with the existing scope
decision.

## Out of scope for this plan (Phase 2b)

- Real Gemini API integration (`geminiProvider.js`)
- Swapping the route's import from `stubProvider` to `geminiProvider`
- Any error handling specific to a real third-party AI call (rate limits, timeouts,
  content-policy failures)
